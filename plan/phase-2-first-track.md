# Phase 2 — PoC: First Track

**Status:** In progress
**Requirements reference:** [docs/requirements.md §8, §9](../docs/requirements.md)

## Goal

Replace the flat plane with a real, hand-built low-poly track — the first proof that the driving feel from Phase 1 holds up on an actual circuit layout, and the first step toward a time-trial game loop.

## Scope

- One hand-built low-poly track, inspired by a real 1990-era circuit's layout/character (corner sequence, elevation), given a fictional name. Built from scratch — no laser scanning or licensed track data.
- Solo time trial: drive laps of the track alone.
- Track-building workflow decided at the start of this phase (open question — see below).

## Out of scope

Lap timing/HUD, sound, opponents, ghost replay, multiple tracks.

## Open questions to resolve at phase start

- Track-building workflow: hand-authored in an external 3D tool and imported, vs. procedurally described in code/data.
- Which real 1990 circuit's character to draw inspiration from for this first track.

**Resolved at phase start (2026-07-17):**

- **Workflow: procedurally described in code (`src/track.ts`)**, not an external 3D tool. A centerline is defined as a small set of hand-placed waypoints (2D position + width); everything else — the smooth spline, road/kerb/grass ribbon meshes, and the matching Rapier collision — is generated from that data. Chosen over an external-tool workflow because it keeps the whole pipeline in the same agentic-coding loop as the rest of the project (no Blender/export step, easy to re-tune corner-by-corner from a plan).
- **Reference circuit: 1990-era Silverstone**, traced from `research/tracks/silverstone/track_layout_all.png` (evolution map, CC BY-SA 4.0, Tom McKay). Specifically the black 1949-1990 base perimeter plus the purple 1987 Woodcote/Luffield alterations (the layout in use through the 1990 season, before the 1991 Priory/Brooklands reprofile). Corner sequence: Copse → Maggotts → Becketts → Chapel → Hangar Straight → Stowe → Vale → Club → Abbey → Bridge → Luffield → Woodcote → pit straight. Waypoint coordinates are hand-estimated from the reference image's proportions (not pixel-traced) — consistent with "hand-built, inspired-by" rather than a laser-scanned recreation, per requirements §8. Flat (no elevation change), matching the real circuit's ex-airfield character and keeping this pass's scope manageable.
- Fictional name: deferred — not yet needed for the driving/collision PoC; can be picked whenever it's first user-visible (HUD, menu).

### 2026-07-17 — Engineering pass complete, human playtest pending

Built `src/track.ts`: a closed Catmull-Rom spline through 28 hand-placed waypoints (see "Resolved at phase start" above for the corner sequence/source), sampled at ~4.6m spacing to generate a flat-shaded asphalt ribbon mesh, curvature-triggered kerb strips (red/white alternating blocks, both inside and outside edge, dilated a bit past the raw curvature threshold so they don't clip on/off abruptly), and a grass plane covering the surrounding area. Collision: one Rapier trimesh matching the paved footprint (track width, widened under kerbs) plus one large flat cuboid for grass, sitting 5cm lower so the raycast always resolves the paved surface first when a wheel is over it. Replaced the Phase 1 flat-plane ground module entirely (`src/ground.ts` deleted, nothing else referenced it).

Grip is now surface-dependent: `VehicleModel` takes a `getSurfaceGrip(colliderHandle)` callback, looks it up per wheel from that wheel's own suspension raycast hit, and multiplies it into the existing grip-circle limit alongside the tire's own `TIRE_GRIP_FRONT/REAR`. Track is tuned to `1.15x`, grass to `0.5x` (see `SURFACE_GRIP_TRACK`/`SURFACE_GRIP_GRASS` in `track.ts`) — first-pass constants, same "needs a human playtest to tune" status as the rest of the grip-circle model. The car also now spawns at the track's start/finish with its heading aligned to the pit straight (`VehicleModel` gained a `spawnYaw` param), instead of always facing world +Z.

Verified via scripted headless Playwright (not a real test suite, matching this project's "no formal tests yet" stance): typecheck/build clean, car spawns on-track with `surfaceGrip=1.15`, accelerates and shifts normally, and — driven deliberately off-line through a turn — grip measurably dropped to `0.5` the instant it crossed onto grass, confirming the surface system actually works end to end rather than just type-checking. Flyby screenshots (via a new `__flyTo` debug hook, same pattern as the existing `__vehicle`/`__getCameraMode` hooks) confirmed the Maggotts-Becketts esses, Stowe, and the Luffield/Woodcote complex all render with correctly-placed kerbs on both sides and asphalt/grass coloring as specified; zero console errors throughout.

What's left, and inherently not agent-completable per the same logic as Phase 1: an actual human lap, to check the corner sequence reads as intended at car speed, whether the kerb curvature threshold/dilation feels right (some corners may show too much/little kerb), and whether the new track/grass grip multipliers need retuning now that there's a real surface to judge them against instead of an open plane.

### 2026-07-17 — Corner distance boards, and a real high-speed roll/pitch instability bug

Added 200m/100m trackside distance-marker boards (pole + red-striped sign, stripe count matching real braking boards) before every corner entry, per a request that they'd help orientation - `findCornerEntries`/`buildDistanceMarkers` in `track.ts`, using the same curvature-derived corner zones as the kerbs.

A second report - the car "still feels pitchy" and rolls too much turning in around 130-160mph, causing instability - turned out to be a real, root-causable bug, not just a tuning gap, matching this project's established pattern (Phase 1 hit several of these). Scripted headless reproduction (teleport to speed on a straight, then accelerate for real from a standstill with zero steering input at all) showed the left/right suspension force could be wildly asymmetric - one side reading a full load, the other reading exactly zero - even driving dead straight, starting as low as ~90mph and getting worse with speed. Root-caused with temporary per-substep force/raycast logging (removed after diagnosis):

- The raw suspension *compression* was fine (sub-millimeter, symmetric) - the anti-roll bars' *stiffness-only* formula, at `FRONT_ARB_STIFFNESS`/`REAR_ARB_STIFFNESS`/`PITCH_ARB_STIFFNESS` = 400000, was discretely unstable at `PHYSICS_SUBSTEPS`'s resolution: left/right suspension force was flipping which side carried the load on *every single substep* (the fastest a discrete sim can alternate at all) - a classic underdamped-spring limit cycle, invisible in telemetry sampled once per outer 60Hz tick because samples happened to land on the same oscillation phase.
- First fix attempt added a rate/damping term to the ARB (the textbook fix for an underdamped spring) - this made it *worse*: the wheel's own raycast compression carries a genuine but tiny (~0.3mm) jitter that already alternates sign every substep on its own, and a damping term turns that into a large, sign-flipping spurious force, saturating the force-transfer cap every substep. A follow-up attempt to smooth that noise via a 2-tap average also made things worse (chased for a bit before verifying against the diagnostic each time), presumably from the extra step of lag reducing stability margin further in an already-marginal discrete feedback loop.
- Fixed by backing the ARB stiffnesses off (400000 → 120000) and doubling `PHYSICS_SUBSTEPS` (4 → 8) - the same two well-precedented levers already used for the clutch/tire stability bounds elsewhere in this file - rather than adding a new derivative term. Re-verified via the same scripted straight-line acceleration run (perfectly symmetric left/right load from a standstill through 160mph+, only rare single-frame blips that self-recover) and a real hard-turn-in run (roll settles to a bounded, physically plausible ~2-3° instead of spiking or oscillating).

Every fix attempt in this pass was verified against the same scripted reproduction before being accepted, including the two that turned out to make things worse - caught by re-running the diagnostic rather than assuming the fix worked.

### 2026-07-17 — Follow-up: inside-wheel lift at high speed still felt disproportionate

Playtesting the fix above found the car still lifts an inside wheel from small steering corrections at 130-160mph, feeling out of proportion to the input. Rather than guess, this was measured with a new scripted diagnostic: teleport to speed, apply a brief (300ms, ramped) steering tap, compare peak roll and wheel loads at 60mph vs 150mph for the *same* input duration.

Two of the user's candidate explanations were tested and ruled out as the dominant cause:

- **Steering-scale curve:** at 150mph the actual steer angle reached was already smaller than at 60mph (3.4° vs 10.2°, the existing speed-sensitive steering doing its job) - yet produced >10x more roll (2.9° vs 0.25°). Halving the high-speed steering ceiling further (`STEER_MIN_FACTOR` 0.18→0.08) only trimmed peak roll to 2.55° - the wheel still fully unloaded either way.
- **Suspension travel:** doubling `SUSPENSION_MAX_TRAVEL` (3.5cm→7cm) didn't help either - it let the chassis lean *further* before topping out, making peak roll slightly worse (3.48°), not better.

The actual mechanism: lateral force demand from a given steer angle scales with vehicle speed (`latSpeed ≈ forwardSpeed * sin(steerAngle)`), and at high speed the aero-boosted grip ceiling is high enough that nearly all of that demand gets through as real force (at low speed the ceiling itself clips most of the demand, which is what was quietly protecting the suspension before). Presented this finding plus four possible fixes to the user (soften the grip ceiling at speed / give the suspension more roll room / keep the lift but fix the recovery / do nothing yet) as a real tradeoff rather than picking one - user chose softening the grip ceiling.

Implemented as `gripRelevantDownforce()`: above `AERO_GRIP_DOWNFORCE_CAP_N` (~120mph's worth of downforce), only `AERO_GRIP_TAPER_ABOVE_CAP` (20%) of additional downforce feeds the tire grip-circle ceiling - the real downforce number (`AERO_DOWNFORCE_K*speed^2`) is untouched and still what telemetry reports; only the grip-relevant portion is capped. Iterated the taper fraction (tried 0.3, 0.1, settled on 0.2) against the same diagnostic. Verified result: peak roll from the 150mph tap dropped from 2.9° to 1.7° (~40%) and the chassis now settles back to level in about half the time - a real, measured improvement, but the inside wheel still fully unloads momentarily from the same input. Explicitly not chasing this further with the same lever: fully eliminating the lift would mean capping the grip ceiling down near its low-speed (no-aero) level, which would flatten out most of the "corners on rails at speed" character the aero model exists to produce in the first place - a genuine tradeoff against the project's own stated 1990-F1 character, not an untried knob. Re-verified numerical stability (the fix from the previous entry) still holds unchanged at every step of this pass.

### 2026-07-17 — Downforce recalibration: 2000kg → 1000kg at 150mph

The car's downforce ceiling had been calibrated (Phase 1) to ~2000kgf at 150mph; flagged as overstating real 1990-era figures. Halved `AERO_DOWNFORCE_K` (4.364 → 2.182) so downforceTotal(150mph) ≈ 9810N (1000kgf), and proportionally halved `AERO_GRIP_DOWNFORCE_CAP_N` (12500 → 6250) so the grip-taper fix above still engages at the same ~120mph threshold rather than silently stopping doing anything now that peak downforce is lower. Updated `docs/requirements.md` §5 to match. Re-verified with the same diagnostics: straight-line numerical stability unaffected, and peak roll from the 150mph steering-tap test dropped further (to ~1.7°, with the low-speed case no longer fully unloading a wheel either) - a welcome side effect of less overall aero grip, not something separately tuned for.

### 2026-07-18 — User tuning pass: chassis stays flat under hard cornering

The user made a direct tuning pass in `vehicle.ts` (not agent-driven): `ROLL_STIFFNESS_NM_PER_RAD` 60000→400000, `ROLL_RATE_DAMPING_NM_PER_RAD_S` 25000→50000, `PITCH_ARB_STIFFNESS` 120000→200000, `STEER_MIN_FACTOR` 0.18→0.10. Reviewed the diff and verified it (typecheck clean, straight-line and hard-turn-in scripted diagnostics from the previous two entries re-run against it: no oscillation reintroduced, numerical stability holds).

Result, confirmed via the turn-in diagnostic: chassis roll under hard cornering at 150mph+ now stays around ~0.3° instead of the previous 2-3°, and stays flat rather than oscillating or spiking - a real, verified improvement in how the car feels from keyboard input. Worth understanding precisely, since it's a bit subtle: `ROLL_STIFFNESS_NM_PER_RAD` is a whole-body torque, separate from the ARBs, that pulls the chassis's roll *angle* back toward level regardless of per-wheel contact state - previously kept deliberately weak (see the ARB entry above) specifically so it couldn't hold the chassis level while a wheel was genuinely airborne. Tuning it back up to real strength means the inside wheel's own suspension force can still fully unload during hard cornering (confirmed unchanged in the diagnostic: `normalLoad` reads 0 for the inside front wheel throughout the corner, same as before) - what's changed is that the chassis itself no longer visibly tilts to reflect that, because the whole-body torque is now strong enough to counteract it independently of the ARB's per-wheel physics. "Flat" and "the inside wheel stops unloading" turned out to be two different things; the user's tuning deliberately favors the former. Comments above the constant were updated to describe this rather than the old "deliberately weak" rationale, including the accepted risk this reintroduces in principle (the torque still doesn't check per-wheel ground contact, so a genuinely-airborne wheel - not exercised by anything in the game yet - could in theory hit the original "floating" bug again).

### 2026-07-18 — Track furniture: perimeter walls, physical curb steps, edge lines

Added the three pieces of track furniture requested: 1.2m perimeter walls both outside the track and around the infield (offset 7m beyond the paved edge, concrete gray by default, switching to an alternating black/white tire-barrier look at corners via the same `isCorner` flag the kerbs use), curbs that physically step up/down 10cm in sync with their red/white coloring (not just a color change), and a permanent 10cm white line down both edges everywhere a curb isn't already marking the boundary.

Key implementation decisions:

- **Walls are real collision, not just visual.** Built as a trimesh (inner + outer vertical faces) alongside the visual mesh, attached to the same fixed track body Rapier already uses for the ground/grass. Verified by teleporting the car at 30 m/s straight at one: it hit, decelerated hard within about a car-length, and came to a dead stop with zero further movement - a real, solid barrier, not a visual-only prop.
- **Curb steps are visual only; collision stays flat.** Same principle already used for the curb's original small paint-height offset: a genuine 10cm bump under a wheel, on a suspension with only 3.5cm of real travel, would risk reintroducing the exact force-spike instability the last two sessions spent fixing. Instead the curb footprint is now a *separate* flat collider (split out of the old single paved-footprint collider) tagged with its own `SURFACE_GRIP_KERB` (0.85 - between the main track's 1.15 and grass's 0.5), so driving over a curb is a real, measured grip penalty without a physical bump risking the suspension. Verified by grid-sweeping wheel positions across a known corner (Becketts) and confirming all three surface grips (1.15/0.85/0.5) appear exactly where expected, in the right order moving from track to curb to grass.
- **Height transitions use riser quads, not a slope.** Each curb block is flat at its own height (raised or flush); where the color/height changes between blocks, a vertical connecting quad closes the gap so it reads as a stepped edge rather than a crack in the mesh or a ramp.
- **Self-intersection risk checked, not assumed.** The inner (infield) wall ring is a perpendicular offset of the same centerline as the outer wall, which only produces valid non-crossing geometry if the offset distance stays below the tightest actual corner radius. Checked visually via top-down screenshots at the two tightest sections (Luffield hairpin, Woodcote chicane) - both rings stay cleanly parallel with no pinching or crossing.
- **A scripted regression check briefly looked broken and wasn't.** Re-running the straight-line, zero-steering high-speed diagnostic from the ARB-stability sessions showed a violent-looking crash partway through. Investigated rather than assumed a physics regression: logging position alongside speed showed the car reaching (1821, 389) right as it broke down - almost exactly the Stowe corner-entry waypoint (1850, 380). That diagnostic never steers, so on a straight line it was always going to run off the track's natural curve eventually; previously there was 400m of open grass to run onto harmlessly, now there's a wall in the way. The wall did its job; the test's own straight-only limitation was what changed, not the vehicle physics - confirmed by re-running the real hard-turn-in diagnostic (which does steer through the corner), which reproduced the same stable ~0.3° roll as the previous entry, unaffected.

### 2026-07-18 — Curb height dropped 10cm → 1cm, collision made real

Follow-up request: the curb step should be small (1cm rise/fall) rather than 10cm, specifically *so* the collision could become physically real (a felt vibration through the car) rather than staying flat - the opposite tradeoff from the original decision, now that the bump is 10x smaller. `KERB_RAISED_HEIGHT` is now `KERB_FLUSH_HEIGHT + KERB_RISE` (0.005 + 0.01) instead of a flat 0.1; `buildKerbColliderDesc` now varies height per block (reusing the same `kerbBlockIndex`/`kerbHeightForBlock` helpers as the visual mesh) instead of a flat strip.

Verified with a precisely-placed diagnostic (not a guessed position - a temporary debug hook exported the real corner-entry sample data (center/right vectors) from `track.ts`, used to place the car exactly straddling a real curb edge with the correct heading, then removed): driving across the curb for real at 15-30mph shows per-wheel suspension load oscillating in a bounded range (~900-1900N front, ~3700-5500N rear against a resting load in a similar range) - a real, felt bump, not a spike or divergence. No change to `SURFACE_GRIP_KERB` (0.85) - the physical bump now works alongside the existing grip penalty rather than instead of it, reinforcing the same "don't use the curb as extra track" goal two ways instead of one.

### 2026-07-20 — Cockpit lowered, curb pushed to 5cm (deliberately less stable this time), aero/tire tuning reviewed

Four requests in one pass:

- **Cockpit + dashboard lowered 20cm** (`COCKPIT_EYE_POSITION.y` 0.95→0.75, dashboard 0.58→0.38, moved together so the dashboard stays in the same place relative to the driver's eye). Feedback was sitting "on" the car rather than "in" it. Confirmed visually via screenshot - the tub sides and wheels now read as properly enclosing the driver instead of sitting below eye level.
- **User-made tuning reviewed, not just accepted blind:** `TIRE_GRIP_REAR` raised above front (1.7→1.85, "rear tires are bigger, lower grip felt wrong") - a reasonable real-world call, comment already updated by the user, nothing further needed. `AERO_DOWNFORCE_K` reduced again (2.182→1.3, "downforce too high for this era") - correct in isolation, but it silently desynced `AERO_GRIP_DOWNFORCE_CAP_N`, the wheel-lift-taper threshold from two entries ago: that constant was a copied-out Newton value calibrated against the *previous* downforce constant to kick in at ~120mph, so leaving it untouched would have silently pushed its effective threshold up to ~155mph instead - the second time a downforce change has desynced it (the first time, halving 2000kgf→1000kgf, was caught and fixed manually). Fixed properly this time: refactored the cap into a formula (`AERO_DOWNFORCE_K * targetSpeedMs^2`) so it always re-targets the same ~120mph regardless of future `AERO_DOWNFORCE_K` changes, instead of needing to be remembered and re-derived by hand every time.
- **Curb rise pushed 1cm → 5cm, explicitly wanting real instability this time**, a reversal of the previous pass's goal (vibration *without* instability). At 5cm the step now exceeds `SUSPENSION_MAX_TRAVEL` (3.5cm), so the wheel genuinely tops out and part of the bump transmits straight into the chassis instead of being fully absorbed - verified via the same precise-placement diagnostic as the previous entry (same real corner location, same methodology): roll swings and load spikes are both visibly larger and rougher than the 1cm version (roll ~1.1-1.6° vs ~0.8° before; peak loads ~7000N vs ~5500N before) while staying bounded - no NaN, no runaway, `SUSPENSION_MAX_FORCE`'s 30000N cap never even approached. Genuinely more unsettling to drive over, not broken.

### 2026-07-20 — Simple time-attack lap timer

Added `src/lap-timer.ts`: a single-checkpoint lap timer (not a full sector/race system, matching the "simple" scope asked for) that detects crossing the start/finish line and tracks current/last/best lap time, displayed as new lines in the existing debug HUD (`Lap: 0:01.600` / `Last: --:--.---  Best: --:--.---`-style formatting). "Crossing" is a sign flip in the car's signed distance along the line's direction of travel (derived from `track.startYaw`), gated to the track's width so the same infinite line-plane can't trigger from clear across the infield on the loop's opposite side.

A real bug surfaced during verification, not assumed away: a scripted test reading lap-timer state ~700ms after page load already showed a "completed" lap that never happened. Root cause: the car spawns sitting exactly on the start/finish line, and the suspension's real settle jitter in the first fraction of a second (the same settling behavior that needed its own clamps in `vehicle.ts` early in this project) rocks the chassis back and forth across the line's threshold before the car has moved at all. Fixed with `MIN_LAP_SECONDS` (10s) - comfortably under any real lap on this ~5.5km track, comfortably over any settle-jitter timescale - so a crossing only completes a lap if the current lap has actually run that long; anything faster is treated as line-adjacent noise and the in-progress lap keeps timing through it. Verified with a corrected scripted test (an earlier draft of the test itself had the track's forward direction backwards, initially making it look like real crossings were being missed - re-derived the direction from `track.ts`'s own startDir computation rather than guessing, which was the actual bug): no phantom lap at spawn, quick spurious re-crossings correctly ignored without disturbing the recorded lap, and a real ~13.4s simulated lap recorded correctly to both last and best.

**Follow-up bug report:** going wide at the end of a lap didn't trigger the timer. The lateral gate had been sized to just the paved track width (`trackWidth/2` = 7.5m), so a car that ran onto the grass verge before the wall - a very normal thing to do, not a wrong-way exploit - fell outside it and the crossing was silently ignored. Widened the gate to span wall to wall: `TrackHandle` now exposes `startLineHalfWidth` (`TRACK_WIDTH/2 + WALL_OFFSET_FROM_EDGE`, computed in `track.ts` from the same constants the walls themselves use, so it can't drift out of sync with where the walls actually are) instead of the old plain `trackWidth`. Verified with two scripted crossings: 12m off-center (beyond the old track-only gate, inside the new wall-to-wall one) now correctly completes the lap; 20m off-center (genuinely beyond the wall) is still correctly ignored - the gate now matches "anywhere you can physically still be," not just "on the painted track."
